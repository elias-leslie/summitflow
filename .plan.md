# Plan: Improve API Response Times for Task List Endpoint

## Objective
Add targeted database indexes to eliminate sequential scans and bitmap-AND anti-patterns in the task list, ready-tasks, and blocked-tasks query paths.

## Current State Analysis

### Existing Indexes on `tasks`
- `idx_tasks_project_status` → composite (project_id, status) — covers dual-filter
- `idx_tasks_project_created` → composite (project_id, created_at DESC) — covers sort
- 12 other single-column indexes (project, status, priority, type, labels=NONE, etc.)

### Existing Indexes on `task_dependencies`
- `idx_task_deps_task` → single (task_id)
- `idx_task_deps_depends` → single (depends_on_task_id)
- Unique constraint on (task_id, depends_on_task_id, dependency_type)

### Performance Gaps Confirmed via EXPLAIN ANALYZE
1. **Labels `@>` filter**: Seq scan on labels column, B-tree cannot accelerate array containment → needs GIN index
2. **Triple-filter (project+status+type or +priority)**: BitmapAnd of separate indexes → needs composite covering index
3. **Dependency subquery**: Seq scan on 26-row `task_dependencies` with filter on (task_id, dependency_type) — acceptable now, but composite index prepares for scale

### Data Profile
- tasks: 90 rows, task_dependencies: 26 rows
- 9 tasks have non-empty labels
- Common query patterns: project+status, project+status+priority, project+labels, ready-tasks dependency check

## Design Decisions

### Why NOT a 4-column composite (project, status, priority, task_type)?
The `list_tasks` query dynamically composes WHERE clauses — not all filters are always present. A `(project_id, status, priority)` 3-column index serves the most common pattern (project+status with priority sort) and the optimizer can combine it with the existing type index via bitmap-AND when needed. A wider index wastes space and only helps the exact 4-filter case.

### Why NOT restructure the EXISTS subquery to NOT IN / anti-join?
EXPLAIN ANALYZE shows PostgreSQL already converts the `NOT EXISTS` to a `Nested Loop Anti Join` — the optimal execution strategy. No query rewrite needed. The composite index on `task_dependencies(task_id, dependency_type)` is sufficient to prepare for scale.

### Why NOT CONCURRENTLY?
All existing migrations use regular `CREATE INDEX IF NOT EXISTS` (not `CONCURRENTLY`). The migration runner (`run_migration.py`) executes within a psycopg3 transaction context and calls `conn.commit()` — `CREATE INDEX CONCURRENTLY` cannot run inside a transaction and would fail. With 90 rows, index creation is sub-millisecond and table locking is negligible.

## Risks
1. **Small dataset may not show optimizer preference changes**: With 90 rows, PG may still choose seq scan over index scan. Validation steps should account for this by using `SET enable_seqscan = off` / `SET enable_bitmapscan = off` to verify indexes CAN be used, then accept that the optimizer makes the right choice at this scale.
